## 1. `singleAdmin` shortcut

### API

Add a single top-level option to `CreateAuthOptions`:

```ts
singleAdmin?: string | string[]
```

```ts
createAuth({
  singleAdmin: "owner@example.com",
  google:  { /* clientId, clientSecret */ },
  passkey: { /* ... */ },
})
```

### Semantics

- String or array of strings; emails are compared case-insensitively
  (`String.prototype.toLowerCase()` on both sides).
- Applied as the default for `magicLink.allowlist` AND `google.allowlist`.
- **Precedence:** if the consumer also passes an explicit
  `magicLink.allowlist` or `google.allowlist`, the explicit callback wins
  for that provider. `singleAdmin` is a shortcut, not a second gate.
  (Decision: a future `singleAdmin: { mode: "and" }` form can extend this
  later if a real need surfaces. YAGNI for now.)
- Google: a matching email with `emailVerified === false` is rejected.
  (Matches the existing recommendation in the Google allowlist docs.)
- Passkey: registration is not separately gated — passkey registration
  requires an existing session, so the magic-link/google gates already
  apply.

### Implementation

- `src/auth/index.ts`: derive `singleAdminAllowlist` and
  `singleAdminGoogleAllowlist` once at the top of `createAuth`.
- Use them as the default for `opts.magicLink?.allowlist` and
  `opts.google?.allowlist` respectively.
- No new files, no new exports.

### Tests

New `tests/single-admin.test.ts`:

- single email, magic-link: matching email is allowed; non-matching is
  rejected (the existing `buildSendMagicLink` returns without sending).
- single email, google: `mapProfileToUser` throws on non-match and on
  `email_verified: false`.
- array form works.
- case-insensitive match.
- explicit `magicLink.allowlist` overrides `singleAdmin` for magic-link
  but not for google.
- explicit `google.allowlist` overrides `singleAdmin` for google but
  not for magic-link.

## 2. `createMiddleware` helper

### API

New subpath: `@naeemba/next-starter/middleware`.

```ts
// middleware.ts at the consumer's repo root
import { createMiddleware } from "@naeemba/next-starter/middleware"

export default createMiddleware({
  protect: ["/admin/:path*", "/dashboard/:path*"],
  signInPath: "/sign-in",        // default
  callbackParam: "callbackUrl",  // default ("next" is also common; configurable)
  cookiePrefix: "better-auth",   // default — matches better-auth's default
})

export const config = { matcher: ["/((?!_next/|favicon.ico|api/auth/).*)"] }
```

### Semantics

- Returns a function `(req: NextRequest) => NextResponse | undefined`.
  Returning `undefined` lets Next continue without modification.
- For each request, if the path matches any `protect` pattern AND the
  session cookie is absent, redirect to `${signInPath}?${callbackParam}=${encodeURIComponent(pathname + search)}`.
- **Check is cookie presence only.** Validating the session requires
  DB access, which won't run in Edge runtime and would force the
  consumer's entire middleware bundle to Node runtime. The "real" gate
  stays at the server-component level via `requireSession` from
  `/server`. Documented as a known limitation.
- Cookie name: `${cookiePrefix}.session_token` (matches better-auth's
  convention). Also checks the `__Secure-` prefixed variant that
  better-auth uses in production over HTTPS.

### Path matching

Lightweight glob → RegExp without pulling in `path-to-regexp`:

- `:param` → `[^/]+`
- `:param*` → `.*`
- `*` → `[^/]*`
- `**` → `.*`
- Other characters escaped.

Patterns are anchored (`^...$`) and compiled once at module load.

### Implementation

- `src/middleware/index.ts`. ~80 LOC.
- Exports `createMiddleware` and the `CreateMiddlewareOptions` type.
- No runtime dep on `better-auth` itself — middleware code path is
  pure cookie/URL manipulation, so it stays Edge-safe.
- Imports `NextRequest` / `NextResponse` from `next/server` (already a
  peer dep).
- tsup entry: add `middleware: "src/middleware/index.ts"` to the
  existing `entry` map; mark `next/server` as external (`next` is
  already external).
- `package.json` exports: add `./middleware` mapping.

### Tests

New `tests/create-middleware.test.ts`:

- request to unprotected path → returns undefined.
- request to protected path with no session cookie → 307 redirect to
  `signInPath` with `callbackUrl` populated.
- request to protected path with session cookie → returns undefined.
- request to protected path with `__Secure-better-auth.session_token`
  → returns undefined.
- custom `cookiePrefix` honored.
- custom `signInPath` honored.
- `:path*` glob matches nested segments.
- multiple `protect` patterns: any match triggers redirect.

Mock `NextRequest` / `NextResponse` from `next/server` rather than
spinning up a Next app (vitest + jsdom is enough; the helpers are
small).

## 3. Optional peers

### Moves

| Package                   | From          | To                                           |
| ------------------------- | ------------- | -------------------------------------------- |
| `postgres`                | dependencies  | peerDependencies (optional) + devDependencies |
| `@react-email/render`     | dependencies  | peerDependencies (optional) + devDependencies |
| `@react-email/components` | dependencies  | peerDependencies (optional) + devDependencies |
| `resend`                  | dependencies  | peerDependencies (optional) + devDependencies |

Stays as hard dependencies (load-bearing on every path):
`better-auth`, `@better-auth/passkey`, `drizzle-orm`, `zod`.

`peerDependenciesMeta` block gets four `{ optional: true }` entries.

### Lazy-load sites

- `src/db/index.ts`: top-level `import postgres from "postgres"`
  becomes a dynamic `await import("postgres")` inside `createDb`.
  `createDb` becomes `async`. *This is a breaking change to the
  function's call signature.* Mitigation: ship a small synchronous
  loader that uses `require()` via `createRequire(import.meta.url)`
  in the ESM build, so `createDb` stays synchronous. The optional-peer
  error message is thrown at the first call site, not at module load.
- `src/email/index.ts`:
  - `render` from `@react-email/render` is only invoked when the
    caller passes `react:` to `sendEmail` OR when the default
    magic-link template runs. Move the `import` to a lazy `loadRender()`
    helper.
  - The default magic-link template imports
    `@react-email/components` from `src/email/templates/magic-link.tsx`.
    Move the template body into a lazily-loaded module so the
    `@react-email/components` import only happens when the consumer
    uses the default template. Consumers passing
    `magicLink.email: customFn` never load it.
- `src/email/resend.ts`: lazy-require `resend` inside `sendViaResend`.

### Friendly error

Shared helper `requireOptionalPeer(name)`:

```ts
function requireOptionalPeer(name: string, hint: string): never {
  throw new Error(
    `[@naeemba/next-starter] Optional peer '${name}' is not installed.\n` +
    `  Install it with:  npm i ${name}\n` +
    `  Used by: ${hint}`
  )
}
```

Called from a `try { return require(name) } catch { requireOptionalPeer(...) }`
wrapper at each lazy-load site.

### Tests

- Reuse existing tests — they pass because the optional peers are still
  in `devDependencies` and resolve normally in the repo.
- Add `tests/optional-peers.test.ts`:
  - mock the module loader to throw `MODULE_NOT_FOUND`, confirm the
    friendly error fires with the right `npm i` hint.

### CHANGELOG / migration

Document under `Changed` (not `Breaking`) because:

- Consumers who already installed via `npm i @naeemba/next-starter`
  have these packages in their lockfile and keep working unchanged.
- Fresh installs see a peer-dep warning if they don't install the
  peers. The warning is exactly the point.

Add a note in UPGRADING.md:

> If you bring your own db client / email template / resend
> alternative, you can omit the matching peer and skip its install
> cost. Otherwise install whichever you use.

## 4. CLI scaffolder

### Invocation

```
npx @naeemba/next-starter init [target-dir]
  --force            overwrite existing files (per-file confirmation otherwise)
  --src              write under src/ regardless of detection
  --no-google        skip the google config in lib/auth.ts
  --no-passkey       skip the passkey config in lib/auth.ts
  --skip-env         do not write .env.example
```

### Layout

```
bin/cli.mjs        published in the tarball
```

`package.json`:

```json
"bin": { "next-starter": "./bin/cli.mjs" },
"files": ["bin", "dist", "README.md", "UPGRADING.md", "CHANGELOG.md", "LICENSE"]
```

### Behavior

- Plain Node, no deps. Uses `node:fs/promises`, `node:path`,
  `node:process`.
- Detects `src/` directory by inspecting consumer's `tsconfig.json`
  `compilerOptions.baseUrl` / paths, falling back to checking
  `src/app/` existence.
- Writes:
  - `lib/auth.ts`
  - `lib/auth-client.ts`
  - `lib/auth-server.ts`
  - `db/schema.ts`
  - `drizzle.config.ts`
  - `app/api/auth/[...all]/route.ts`
  - `app/sign-in/page.tsx`
  - `.env.example`
- Per-file: skip if exists, unless `--force`. Print a one-line summary
  at the end:
  - green `+` for written
  - yellow `=` for skipped (already present)
  - red `!` for forced-overwrite
- Final block: a checklist of next steps (env vars to set, db migrate
  command, etc).

### Templates

Inlined as JS template strings at the top of `bin/cli.mjs`. The seven
files are short; a single ~250-line CLI is much easier to maintain than
a separate templates directory.

### Tests

`tests/cli.test.ts`:

- spawn `node bin/cli.mjs init <tmpdir>` and assert each file exists
  with non-empty content.
- second run without `--force` reports "skipped" for all and writes
  nothing.
- `--no-google` omits the google block.
- exits non-zero on a malformed target path.

## Non-goals

- No `middleware` runtime config opinion. The consumer owns `matcher`.
- No interactive prompts in the CLI v1; flags only. Adding `prompts` /
  `enquirer` would force a dep.
- No automated `migration` from 0.3.x deps to peers. Consumers already
  using these packages keep them; the peer warning surfaces on fresh
  installs.
- No `createAuthClient` shortcut for `singleAdmin` — clients don't
  enforce auth.

## Risks

- Lazy-loading `postgres` in `createDb`: the dynamic require has to
  succeed synchronously to keep `createDb` synchronous. Using
  `createRequire(import.meta.url)` from `node:module` works in ESM
  but throws in non-Node runtimes. Acceptable: the package is already
  Node-only (Next server runtime).
- `next/server` types in middleware tests: the project already pulls
  `next` as a devDep, so types are available. Run-time mocks for
  `NextRequest`/`NextResponse` are tiny.
- CLI on Windows: paths and shebangs. Use `path.join` everywhere;
  `#!/usr/bin/env node` works on Windows too via the npm shim.
